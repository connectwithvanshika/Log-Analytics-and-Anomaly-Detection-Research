# Step 1: Data Collection Workflow (No AI/ML Involved)

Before Dynatrace can perform anomaly detection, correlation analysis, or root cause identification, it must first collect data from every component of the technology stack. This initial stage is known as **Data Collection** and is handled primarily by **Dynatrace OneAgent**.

At this stage, no Artificial Intelligence (AI) or Machine Learning (ML) is involved. The objective is simply to gather raw telemetry data and establish visibility across the entire environment.

---

## Step 1.1: Deploy OneAgent

Organizations install Dynatrace OneAgent on their infrastructure, including:

* Physical servers
* Virtual machines
* Cloud instances
* Containers
* Kubernetes clusters
* Databases
* Application servers

Once installed, OneAgent begins monitoring the environment automatically without requiring manual instrumentation.

```text
Applications
Servers
Containers
Databases
Cloud Resources
        ↓
     OneAgent
```

---

## Step 1.2: User Interacts with the Application

Consider an e-commerce application.

A customer visits the website and clicks **"Buy Now."**

```text
Customer
    ↓
Website
    ↓
Backend API
    ↓
Database
```

This action triggers multiple interactions across different services and infrastructure components.

OneAgent observes every step of this transaction.

---

## Step 1.3: Collect Application Telemetry

OneAgent attaches itself to application processes and captures information such as:

* Incoming requests
* Outgoing responses
* Response times
* Service dependencies
* Exceptions and errors

Example:

```text
GET /buy-product
```

The agent records:

* Request received
* Processing time
* Response generated
* Services involved

This information is used to build **distributed traces**.

Example:

```text
User Request
      ↓
Frontend Service
      ↓
Payment Service
      ↓
Database
```

This allows Dynatrace to understand the complete execution path of a request.

---

## Step 1.4: Collect Infrastructure Metrics

In addition to application data, OneAgent continuously monitors the underlying infrastructure.

Examples include:

```text
CPU Usage = 92%
Memory Usage = 85%
Disk Usage = 78%
```

Metrics collected include:

* CPU utilization
* Memory consumption
* Disk I/O
* Network traffic
* Process health
* Resource availability

These measurements provide visibility into infrastructure performance.

---

## Step 1.5: Collect Logs

Applications constantly generate logs describing events and system behavior.

Examples:

```text
INFO: User Login Successful

WARNING: Slow Payment Response

ERROR: Database Connection Timeout
```

OneAgent collects these logs and forwards them to the Dynatrace platform.

Logs provide detailed information about application activity and failures.

---

## Step 1.6: Monitor Databases

Databases are often a critical dependency in modern applications.

OneAgent monitors:

* Query execution time
* Failed queries
* Connection issues
* Database resource utilization

Example:

```sql
SELECT * FROM orders;
```

Normal execution time:

```text
50 ms
```

Abnormal execution time:

```text
5000 ms
```

Such observations are collected for later analysis.

---

## Step 1.7: Monitor Kubernetes and Containers

Modern cloud-native applications typically run inside Kubernetes environments.

Example:

```text
Kubernetes Cluster
│
├── Frontend Pod
├── Payment Pod
└── Database Pod
```

OneAgent automatically discovers:

* Clusters
* Nodes
* Pods
* Containers
* Services
* Deployments

It collects metrics such as:

* Container CPU usage
* Container memory usage
* Pod restarts
* Pod failures
* Container health status

This provides complete visibility into containerized environments.

---

## Step 1.8: Monitor User Experience

Dynatrace also collects real-user monitoring data.

Examples include:

* Page load time
* User interactions
* JavaScript errors
* Session duration
* Frontend performance

Example:

```text
Home Page Load Time = 7 Seconds
```

Even if servers appear healthy, Dynatrace can detect poor user experiences.

---

## Step 1.9: Automatic Topology Discovery

As data is collected, Dynatrace automatically identifies relationships between components.

Example:

```text
User
  ↓
Frontend Service
  ↓
Payment API
  ↓
Database
```

Dynatrace discovers:

* Which service communicates with another service
* Which container hosts an application
* Which database supports a service
* Which Kubernetes resources belong together

This creates a real-time dependency map known as **Smartscape**.

---

## Step 1.10: Send Telemetry Data to Dynatrace Platform

After collection, all telemetry data is transmitted to the Dynatrace platform.

```text
Applications
Servers
Containers
Databases
Logs
Metrics
Traces
User Experience Data
         ↓
      OneAgent
         ↓
 Dynatrace Platform
```

At this stage:

* No AI is being used.
* No machine learning is being used.
* No anomaly detection has occurred.
* No root cause analysis has occurred.

Dynatrace is simply collecting and organizing telemetry data from the environment.

---

## Why Dynatrace Collects Multiple Types of Data

Problems can originate from different layers of the technology stack.

Examples:

### Infrastructure Issue

```text
CPU Usage = 95%
```

### Database Issue

```text
Database Query = 10 Seconds
```

### Application Issue

```text
Payment API Failure
```

### User Experience Issue

```text
JavaScript Error on Checkout Page
```

If Dynatrace collected only logs, it would miss infrastructure problems.

If it collected only metrics, it would miss application failures.

If it collected only traces, it would miss user experience issues.

Therefore, Dynatrace collects:

* Logs
* Metrics
* Traces
* Events
* User Experience Data
* Infrastructure Data

to create a complete and contextual view of the system.

---

## Summary

Data Collection is the foundation of the Dynatrace platform. Using OneAgent, Dynatrace continuously gathers telemetry data from applications, infrastructure, databases, containers, Kubernetes clusters, cloud services, and end users. It automatically discovers dependencies between components and builds a contextual view of the environment. This comprehensive dataset serves as the input for later stages such as baseline learning, anomaly detection, correlation analysis, and root cause identification.
